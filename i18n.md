# Internationalization (i18n)

Psychic applications support two complementary i18n patterns:

1. **Code-driven i18n** - Translate static values (enum labels, UI strings) using locale files and `I18nProvider`
2. **Data-driven i18n** - Translate user-generated content (titles, descriptions) using a polymorphic `LocalizedText` model stored in the database

Both patterns use the `Accept-Language` request header to determine the locale, and pass it through to serializers via `serializerPassthrough`.

## Code-Driven i18n (I18nProvider)

Use this for translating enum values, labels, and other static strings that are defined in code rather than entered by users.

### Setup

**1. Create locale files** in `src/conf/locales/`:

  rooms: {

        twin: 'twin',

**2. Create locale index** that maps locale keys to locale files:

```typescript
// src/conf/locales/index.ts
import en from '@conf/locales/en.js'
import es from '@conf/locales/es.js'

export default {
  en,
  es,
}
```

**3. Create the i18n utility** using `I18nProvider` from Psychic:

```typescript
// src/utils/i18n.ts
import locales from '@conf/locales/index.js'
import { I18nProvider } from '@rvoh/psychic/system'
import { LocalesEnumValues } from '@src/types/db.js'

export function supportedLocales() {
  return LocalesEnumValues
}

export default I18nProvider.provide(locales, 'en')
```

The second argument to `I18nProvider.provide` is the default locale prefix. Locale enum values like `'en-US'` are matched to locale keys by prefix (e.g., `'en-US'` maps to the `en` locale file).

**4. Tie supported locales to the database enum** so locales are defined in one place (the migration) and flow through the type system.

### Using i18n in Serializers

Pass the locale through `serializerPassthrough` and use `i18n()` in `customAttribute`:

```typescript
import i18n from '@src/utils/i18n.js'
import { LocalesEnum } from '@src/types/db.js'

export const PlaceForGuestsSerializer = (place: Place, passthrough: { locale: LocalesEnum }) =>
  DreamSerializer(Place, place)
    .attribute('id')
    .attribute('style')
    .customAttribute('displayStyle', () => i18n(passthrough.locale, `places.style.${place.style}`), {
      openapi: 'string',
    })
```

For enum arrays (e.g., bed types), use `ObjectSerializer` to translate each value:

```typescript
export const BedTypeSerializer = (bedType: BedTypesEnum, passthrough: { locale: LocalesEnum }) =>
  ObjectSerializer({ bedType }, passthrough)
    .attribute('bedType', { as: 'value', openapi: { type: 'string', enum: BedTypesEnumValues } })
    .customAttribute('label', () => i18n(passthrough.locale, `rooms.Bedroom.bedTypes.${bedType}`), {
      openapi: 'string',
    })

export const RoomBedroomForGuestsSerializer = (bedroom: Bedroom, passthrough: { locale: LocalesEnum }) =>
  RoomForGuestsSerializer(Bedroom, bedroom, passthrough)
    .rendersMany('bedTypes', { serializer: BedTypeSerializer })
```

### Passing Locale from Controllers

Read the `Accept-Language` header in a base controller and pass it via `serializerPassthrough`:

```typescript
export default class AuthedController extends ApplicationController {
  @BeforeAction()
  public configureSerializers() {
    this.serializerPassthrough({ locale: this.locale })
  }

  protected get locale() {
    const locale = this.header('Accept-Language')
    const locales = supportedLocales()
    return locales.includes(locale as (typeof locales)[number]) ? locale : 'en-US'
  }
}
```

When querying models that use passthrough-conditioned associations (like `currentLocalizedText`), pass the locale on the query:

```typescript
const places = await Place.passthrough({ locale: this.locale })
  .preloadFor('forGuests')
  .all()
```

### Output encoding boundary

`I18nProvider` performs literal-substring substitution on `%{name}` placeholders â€” **interpolated values are passed through verbatim, NOT HTML-escaped**. This is correct because Psychic is a JSON API. The encoding flow is:

1. **Server side (automatic):** `JSON.stringify` escapes `"`, `\`, and control characters when emitting the response, so a translated string containing `<script>alert(1)</script>` arrives at the client as the literal seven-character substring `<script>` followed by the rest. JSON-safe in transit.
2. **Client side (renderer's job):** when the client inserts the string into a DOM, the rendering framework escapes it for the right context. React's `{value}`, Vue's `{{ value }}`, and Svelte's `{value}` all HTML-escape by default. That's where XSS prevention actually lives.

**Don't add `.replace(/</g, '&lt;')` (or any other defensive HTML escape) to translation values on the server.** It breaks non-HTML consumers (React Native `<Text>`, CLI tools, downstream API consumers, PDF generators) which would receive `&lt;script&gt;` and have to un-escape before re-encoding. It also under-protects: HTML-body escaping doesn't address attribute, URL, or JS-string contexts that the same value might land in. Context-aware escaping happens at DOM insertion, not at translation lookup.

## Data-Driven i18n (LocalizedText)

Use this for user-generated content that needs to be translated - titles, descriptions, markdown content, etc. This pattern uses a polymorphic `LocalizedText` model that can belong to any model needing localized content.

### LocalizedText Model

A polymorphic model with `localizableType`/`localizableId` columns and a `locale` column:

```typescript
@SoftDelete()
export default class LocalizedText extends ApplicationModel {
  public override get table() { return 'localized_texts' as const }

  public id: DreamColumn<LocalizedText, 'id'>
  public localizableType: DreamColumn<LocalizedText, 'localizableType'>
  public localizableId: DreamColumn<LocalizedText, 'localizableId'>
  public locale: DreamColumn<LocalizedText, 'locale'>
  public title: DreamColumn<LocalizedText, 'title'>
  public markdown: DreamColumn<LocalizedText, 'markdown'>

  @deco.BelongsTo(['Host', 'Place', 'Room'], { polymorphic: true, on: 'localizableId' })
  public localizable: Host | Place | Room
}
```

### Associating LocalizedText with Models

Each model that needs localized content gets two associations:

```typescript
// HasMany for all localized texts (used by hosts for CRUD)
@deco.HasMany('LocalizedText', { polymorphic: true, on: 'localizableId', dependent: 'destroy' })
public localizedTexts: LocalizedText[]

// HasOne with passthrough condition for the current locale (used by guests for display)
@deco.HasOne('LocalizedText', {
  polymorphic: true,
  on: 'localizableId',
  and: { locale: DreamConst.passthrough },
})
public currentLocalizedText: LocalizedText

// HasOne fixed to the always-present default locale, used as a fallback when the
// requested locale has no LocalizedText row (see "Falling Back to a Default Locale" below)
@deco.HasOne('LocalizedText', {
  polymorphic: true,
  on: 'localizableId',
  and: { locale: 'en-US' },
})
public fallbackCurrentLocalizedText: LocalizedText
```

The `DreamConst.passthrough` condition means the `locale` value comes from the query's passthrough context at query time, not from a fixed value. A model can have multiple `HasOne` associations to the same target with different `and:` conditions â€” one passthrough-driven, one fixed â€” and both resolve independently in the same preload tree.

### Creating Default LocalizedText

Use an `@AfterCreate` hook to create a default localized text when the parent is created:

```typescript
@deco.AfterCreate()
public async createDefaultLocalizedText(this: Place) {
  await this.createAssociation('localizedTexts', { locale: 'en-US', title: `My ${this.style}` })
}
```

### Using in Serializers

Use `delegatedAttribute` to pull a field from the loaded `currentLocalizedText`. Because
the requested locale may have no `LocalizedText` row, layer the fixed-locale
`fallbackCurrentLocalizedText` association (added above) underneath it, so the field
falls back to the always-present default-locale translation instead of going blank â€”
since `@AfterCreate` guarantees an `en-US` row (see above), a Host can save a partial
translation for one locale without blanking the field in every other locale:

```typescript
export const PlaceSummaryForGuestsSerializer = (place: Place) =>
  DreamSerializer(Place, place)
    .attribute('id')
    .delegatedAttribute('fallbackCurrentLocalizedText', 'title', { openapi: 'string' })
    .delegatedAttribute('currentLocalizedText', 'title', { openapi: 'string', required: false })
```

This is the general "default value, optionally overridden" technique â€” see
[serializers.md â€” Layering two `delegatedAttribute`s onto the same output
key](serializers.md#layering-two-delegatedattributes-onto-the-same-output-key) for why
the second call uses `required: false` rather than `optional: true`.

### Querying with Locale Passthrough

Pass the locale so `currentLocalizedText` resolves to the correct locale's record:

```typescript
const place = await Place.passthrough({ locale: this.locale })
  .preloadFor('forGuests')
  .findOrFail(this.castParam('id', 'uuid'))
```

### Migration

The LocalizedText table needs a unique index on `(localizable_type, localizable_id, locale)` to ensure one translation per locale per record:

```typescript
await db.schema
  .createIndex('localized_texts_localizable_for_locale')
  .on('localized_texts')
  .columns(['localizable_type', 'localizable_id', 'locale'])
  .unique()
  .execute()
```

## Combining Both Patterns

The two patterns are complementary:

- **Code-driven** (`i18n()`) translates enum values and static labels - things that are the same for every user (e.g., "cottage" -> "cabaÃ±a")
- **Data-driven** (`LocalizedText`) translates user-generated content - things that are unique per record (e.g., a place's title or description)

A single serializer often uses both:

```typescript
export const PlaceForGuestsSerializer = (place: Place, passthrough: { locale: LocalesEnum }) =>
  // Data-driven: PlaceSummaryForGuestsSerializer already delegates `title` from
  // currentLocalizedText, falling back to fallbackCurrentLocalizedText
  PlaceSummaryForGuestsSerializer(place)
    .attribute('style')
    // Code-driven: translate the style enum
    .customAttribute('displayStyle', () => i18n(passthrough.locale, `places.style.${place.style}`), {
      openapi: 'string',
    })
    .rendersMany('rooms', { serializerKey: 'forGuests' })
```

## Combining STI with LocalizedText

STI can be combined with the polymorphic `LocalizedText` pattern so that different content types get different localized text shapes while sharing the same table. For example, a `LocalizedVideoText` STI child could add a `transcript` field that `StandardLocalizedText` does not have:

```typescript
// Base: shared columns (title, markdown, locale, etc.)
export default class LocalizedText extends ApplicationModel {
  public type: DreamColumn<LocalizedText, 'type'>
  // ... shared columns
}

// STI child: adds video-specific fields
@STI(LocalizedText)
export default class LocalizedVideoText extends LocalizedText {
  public transcript: DreamColumn<LocalizedVideoText, 'transcript'>

  public override get serializers(): DreamSerializers<LocalizedVideoText> {
    return {
      default: 'LocalizedVideoTextSerializer',
      summary: 'LocalizedVideoTextSummarySerializer',
    }
  }
}
```

The `LocalizedVideoTextSerializer` would include the `transcript` attribute, while the base `LocalizedTextSerializer` would not. Since STI and polymorphic associations both operate on the same table, models that need video localization can associate with `LocalizedVideoText` while others continue using the base `LocalizedText`.
